我天天说「循环 > 派活」,
其实我只是
派了一次活.

「循环 > 派活」是我给自己写进操作系统的一条信条。可读完 Anthropic 这篇《Getting started with loops》我才发现:我把 loop 挂在嘴上,身体却很诚实——大多数时候,我只是对着 AI 派一次活、收一次货,谈不上循环。文章把 loop 分成四级阶梯,而真正戳中我的是那条暗线:每升一级,你就多交出去一件事——验证 → 停止条件 → 触发 → 连提示都不用。这不就是我一直在写的「把人安全地移出信任链」,只是这次终于有了能一级一级踩上去的梯子。

先搞清楚 · Loop 是什么

让 agent 一圈圈转,
直到达标才停

文章给 loop 的定义朴素得很:agent 反复循环地干活,直到满足一个「停止条件」为止。区别只在四件事上——怎么被触发、靠什么停、用哪个 Claude Code 原语、适合哪类活。

核心心法就一句:把重复干、又能定义「怎么算做完」的活,交给循环。但文章也没让你上头——最后收在一句「不是所有活都需要复杂的循环,先用最简单的方案」。这和我做事的克制正好对味:别为了循环而循环。

它的四级阶梯,我读出了一条原文没明说、但我特别在意的暗线:这不是四种并列的工具,是一部电梯。你每往上升一级,就把一件本来攥在自己手里的事,交给了机器——我把它按「你交出去了什么」重排了一遍:

四级阶梯 · 每升一级,多交出去一件事

一部把你
移出信任链的电梯

1
你交出去:验证

手动循环Turn-Based

你 prompt 一次,Claude 收集上下文 → 干 → 验证 → 回你,然后停,等你下一句。升级点:把「验证」写成 skill——比如一个自动起 dev server、点一遍改动、看浏览器 console 才交付的验证 skill,省得你每次手动检查。

2
你交出去:停止条件

目标循环Goal · /goal

你不再一句句盯,而是定义「什么算成功」,它自己转到达标或到上限才停。例:/goal 把首页 Lighthouse 跑到 90 以上,试 5 次就停。关键是把成功条件和上限写清,它才不会半路宣布搞定。

3
你交出去:触发

定时循环Time · /loop · /schedule

连「什么时候开始」都不用你了。/loop 5m 查我的 PR、处理评审意见、修挂掉的 CI 在本地按间隔重跑(关机就停);/schedule 把它挪到云端当例行任务。心法:间隔对齐真实变化频率,或干脆按事件触发,别空转烧 token。

4
你交出去:连提示都不用

自主循环Proactive

没有实时的人参与。把 /schedule + /goal + skills + dynamic workflows + auto 模式拼起来,让它长期跑一条定义清楚的活儿线:自动查反馈 → 三分类 bug → 并行探方案 → 对抗式 code review。适合 bug triage、迁移、依赖升级这种「反复且明确」的流。

看清这条暗线之后,上一篇(dynamic workflows)和这一篇就接上了:workflow 是「一次把活拆开」,loop 是「把这件事一直干下去」。而第 4 级自主循环,恰好是把 skill、workflow、goal 全拼进一部一直转的机器——我那三篇读的,其实是同一台机器的三个零件。

上手术台 · 我实际站在第几级

信条第 4 级,
身体第 1 级

拿这部电梯量我自己的舰队和日常,结果有点难堪:我嘴上信「循环 > 派活」,身体却基本住在一楼。

级别
我住在这吗
证据 / 缺口
Lv1 · 手动循环
✓ 全都是
我整支舰队几乎都在这一级:我说「盘一盘」它跑一次、我说「切到 Replen」它答一次。而且我连 Lv1 的升级都没做全——没几个 skill 带自动验证 check。
Lv2 · 目标循环 /goal
✗ 几乎没
我很少写「什么算做完 + 试几次就停」。方案通常一稿交,靠我肉眼判断够不够好——停止条件一直攥在我自己手里。
Lv3 · 定时循环 /loop /schedule
半分
我知道 /loop、也偶尔用,但没把任何一条真实工作流挂成按间隔自动跑的例行任务。触发这件事,基本还是我手动扣扳机。
Lv4 · 自主循环
就一条
唯一的例外:Chief of Staff 每天早晨自动出 briefing 那条云端定时任务——这是我全舰唯一一台「不用我开口」的机器。证明我做得到,只是只做了一台。

诊断很清楚:我不是不懂循环,是把电梯当楼梯用——每次都亲自爬,爬完就下来。三篇连起来看,毛病是同一个形状:skill 缺「坑」、舰队缺「唱反调的」、日常缺「一直转的循环」——全是「交出去」这一步没做够。

动手 · 挑一件事,升一级

别贪四级,
先升一级就好

文章的上手法很实在:找一件你反复做的瓶颈活,问三个问题——验证 check 你写得出来吗?目标够明确吗?活是不是按时间到的?然后启动、看它在哪卡壳 / 哪越界、再迭代。我按这个,给自己挑了一件最该升级的:

选题 · 从 Lv1 → Lv3

把「投资简报」从手动派活,变成每周定时循环

我现在是想起来才让 CIO 盘一次组合。它完美符合三个问题:验证 check 写得出(mark-to-market 有没有更新、有没有触发段永平纪律的阈值)、目标明确(出一份简报)、按时间到(每周一次足矣)。

Step 1 · 先补 Lv1 的验证

给它一个「够不够好才交」的自查 skill

先别急着定时。按文章的顺序,先把验证写成 skill:每份简报交付前自查「数据是否 mark 到最新、每条结论有没有证据、有没有漏掉的持仓」。验证稳了,循环才敢放手。

Step 2 · 交出停止条件(/goal)

写清「什么算一份合格简报」

用 /goal 定义达标标准 + 上限(比如「三条以内高信号变动、每条附证据,迭代不超过 3 轮」),让它自己转到够好为止,而不是等我肉眼点头。

Step 3 · 交出触发(/schedule)

挂成每周云端例行,间隔对齐真实频率

最后用 /schedule 把它挪上云端、每周跑一次——就像 Chief of Staff 那条 briefing。组合还能再叠上上一篇的对抗验证当第 4 级的一环:简报出稿后,派个红队 agent 挑刺再交我。

注意我没有一步登天去搭第 4 级。文章那句克制我记住了:先用最简单的方案。挑一件事、升一级、看它卡在哪再补——这本身就是一个 loop。

我一直把电梯,
当楼梯在爬.

「循环 > 派活」我信了很久,但信条不落地就只是一句漂亮话。这篇给我的不是新道理,是一部能一级一级踩上去的梯子:每升一级,我就把一件事——验证、停止条件、触发、乃至提示——诚实地交出去一次。而每交出去一次,我就离自己那句「把人安全地移出信任链」近一步。

交出去的
每一件事,
都得先值得信.

关键词是安全地移出。上一篇教我给舰队补「唱反调的人」,正是为了这一篇:你敢把验证、停止条件、触发一层层交出去,前提是有一道对抗式的自查在兜底。信任不是省出来的,是一级一级验出来的。这三篇——skill 补坑、舰队补证伪、日常补循环——其实是同一台机器的三个零件:一台我能放心让它替我一直转下去的机器。

下一步很具体:把投资简报那条,从 Lv1 真的升到 Lv3。这篇的下一篇,应该是它第一次没等我开口、就把简报放我桌上的那个早晨。
